# Apollo Contributor 가이드

아폴로에 흥미가 있고 더 나아지기를 바라세요? 우리도 그래요!

아폴로는 당신과 같은 개발자들이 모인 커뮤니티이고, GraphQL에 관한 최고의 툴과 라이브러리들을 만드는 노력을 하고 있어요. 우리는 나이나 경험에 상관없이 기여와 건설적인 피드백을 제공해주는 누구든 환영해요! 만약 여러분께서 도움을 원하지만, 어디서 시작해야 할지 막막하다면 우리에게 알려주세요. 우리는 여러분을 위해 해결책을 찾을 거에요.

오, 그리고 혹시 아직 다음 채널에 없다면 가입하시길 바래요. [Apollo Slack](https://www.apollographql.com/#slack).

아래에 프로젝트에 기여하기 가장 쉬운 것부터 가장 어려운 것까지 몇가지 방법이 있어요.

## Issues

### 버그 제보

만약 버그를 만나게 된다면, 버그가 포함되어 있다고 생각되는 하위 프로젝트의 repository를 통해 Github 이슈를 추가해 주세요. 만약 여러분이 가진 이슈가 이미 제보되어있다면, 이슈에 대한 동의를 나타내기 위해 추가적인 정보나 👍을 표현해주세요.
우리가 보고받은 모든 이슈에 대해 도움을 드리는 동안, 최대한 빠르게 수정할 수 있도록 하기 위해 다음을 포함해주세요.

1.  **의도된 결과:** 여러분이 버그가 발생 했을 때 목표로 했던 것, 그리고 문제와 관련된 가능한 많은 코드를 남겨주세요
2.  **실제 결과:** 실제 무슨 일이 일어 났는지에 대한 설명, 관련된 에러 메시지를 복붙하거나, 스크린샷 포함시키기, 에러 로그, 혹은 그 밖의 관련된 결과를 알려주세요. 여러분의 브라우저 콘솔, 서버 콘솔, 그리고 네트워크 로그와 같은 것도 알려주셔야 할 정보에 포함됩니다. “didn’t work” or “broke” 와 같은 명시적이지 않은 문장들은 피해주세요.
3.  **어떻게 이슈를 재현하는지:** 보수를 하는 분과 기여자분들에게 어떻게 이슈가 재현될 수 있는지 알려주세요. 가능한 구체적으로, 그리고 그 버그를 재현할 수 있는 필수적인 것들만 알려주세요. 만약 가능하다면, 버그가 발생하는 정확한 상황을 분리해주시고, 발생한 원인을 추측하는 것은 피해주세요.
    좋은 재현을 만드는 것은 기여자분들에게 여러분들의 이슈를 빠르게 조사하고 해결하는데 정말 큰 도움이 됩니다. 많은 케이스에서, 최소한의 재현을 구현하는 행위들은 문제가 있는 라이브러리 바깥에서 버그의 원인이 있다는 것을 알려주기 때문에 모두에게 시간과 노력을 아껴 줄 것입니다.

### documentation 향상

문서 향상, 예제, 그리고 다른 오픈 소스 내용과 같은 것은 라이브러리에 기여하기 위한 가장 쉬운 방법 입니다.
만약 여러분께서 발견한 것이 아주 작은 내용이라도 이것이 더 나아질 수 있는 방안이라면, Pull Request를 여세요!
만약 여러분이 중요하고 더 큰 변화를 제안하고 싶다면, 우리는 여러분의 아이디어를 듣고 싶지만, 일단은 PR을 작성하기 전에 토론할 수 있는 이슈를 열어주세요.

### 이슈 응답

이슈를 보고하는것 이외에도, 다른 사람들의 이슈를 응답하고, 문제를 식별하고, 혹은 그들의 작업을 돕는것이 Apollo에 기여하는 가장 좋은 방법일 것입니다. 만약 여러분이 이 프로세스에서 더 활동적인 역할을 하는 것에 관심이 있다면, 이슈에 계속해서 응답해주세요. 그리고 Apollo Slack에 "Hi" 라고 말하는것 잊지 마세요!

### 작은 버그 처리

작은 버그 처리 변경을 위해서 (20줄 미만의 코드 변경), 자유롭게 pull request를 여세요. 우리는 가능한 빨리 그것을 merge하려고 시도 할 것이고, 같은날 새로운 릴리즈를 게시하는 것이 이상적일 것입니다. 단지 저희의 요구사항은, 여러분이 고치려고 시도한 버그를 검증하기 위한 테스트를 추가하는 것을 확실히 해주세요.

### 기능 제안

Apollo의 대부분의 기능은 커뮤니티의 여러분들의 제안으로부터 만들어졌어요! 우리는 여러분이 사용하는 케이스에서 어떻게 Apollo가 발전할 수 있을지에 대한 어떤 아이디어든 환영해요. 기능에 대해 전폭적인 수요가 없다면 그것이 즉시 구현되기는 힘들수 있지만, 사람들이 여러분의 제안에 대해 토론 할수 있도록 가능한 많은 정보를 포함해주세요.

1.  **사용 사례:** 구체적으로 무엇을 성취하고 싶으세요? 보통, 이미 여러분이 필요한 좋은 방법들이 있으며, 새로운 기능이 필요 없을지도 있지만, 구체적인 사용 사례에 대한 정보 없이는 어떤 것인지 알기 어려워요.
2.  **Plugin일까요?** 많은 사례에서, 라이브러리의 핵심에 포함하기에는 기능이 아주 일부분만 관련이 있어서, 동반된 패키지와 더 잘 구현된 경우가 있어요. 만약 라이브러리를 확장하여 원하는 작업을 할 수 있는 방법이 없는 경우 우리가 plugin API들을 추가할 수 있나요? 왜 기능이 라이브러리의 핵심 기능의 한 부분이 되어야 하는지 이해하기 위해 사례를 만드는 것이 중요해요.
3.  **해결책이 있나요?** 이미 가능한 해결책들보다 더 편한 방법이 있나요? 혹은 해결책을 불가능하게 만드는 방해물들이 있나요?

기능 요청은 다음과 같이 표시 되고, 우리는 새로운 기능과 가능한 구현 디자인을 토론하기 위한 장소로 Github issues를 사용하는 것을 격려하고 있어요.
기능이 포함되어야 한다는 합의가 있기 전까지는 제안된 기능을 구현하기 위한 pull request를 제출하는 것을 삼가해주세요. 이 방법은, 여러분이 merge할 수 없는 작업을 하는 것을 피할 수 있습니다.

새로운 기능이 필요하다는 합의가 이루어지면, 아래의 "Big PRs" 리스트를 진행하세요.

## Big PRs

다음을 포함하세요:

- 큰 버그 수정
- 새로운 기능들

repository의 중요한 변화를 하기 위해서, 구현을 시작하기 전에 디자인을 결정하는 것이 중요합니다. 이런 방법이, 우리가 중요한 개선사항들이 받아야할 관심과 관리를 받아야 한다는 것을 확신 할 수 있게 합니다. 큰 변화는 위험을 동반할 수 있고, 항상 merge될수는 없기 때문에, 이것은 먼저 디자인/계획의 구현에 대한 동의를 함으로써 가능한 낭비되는 노력의 양을 줄이는데 좋은 방법일 것입니다.

1.  **이슈를 여세요.** 위에서 설명했듯이, 버그 혹은 기능에 대한 이슈를 여세요
2.  **합의에 도달하세요.** 몇몇 기여자들과 커뮤니티 멤버들은 이 기술 혹은 버그가 중요하다는데 동의해야 합니다, 그리고 그들 중 누군가는 구현하고 문제를 해결해야 해요.
3.  **의도 된 행동을 합의하세요.** 이슈에 대해, 의도된 행동에 대한 합의에 도달하세요. 버그를 고치는 경우에, 수정 되기 위한 버그에 대한 의미가 명확해야 하고, 기능 개선의 경우, 개발자가 사용하는 새로운 기능이 어떤 것인지 명확해야 합니다.
4.  **구현 계획에 대해 합의하세요.** 어떻게 기능 개선 혹은 버그 수정이 구현되어야 할지에 대한 계획을 작성하세요. 어떤 모듈이 추가되어야 하고, 재작성 되어야 하나요? 이것이 한번 pull request를 하면 되나요? 혹은 몇번이고 개선해가야 할 사안인가요? 각각의 파트를 누가 할 것인가요?
5.  **PR을 제출하세요.** 변경사항을 구현하기 위해, 여러개의 종속 patch들을 만들어야 하는 경우, 한번에 하나씩만 제출해주세요. 그렇지 않으면, 첫번째가 review와 merge되는 동안, 나머지 것들이 시간이 지남으로써 문제가 발생할 수 있습니다. “while we’re here” 같은 유형의 변경을 피하는 것을 명심해주세요 - 만약 현재 개선과 관련없는 것이 있다면, 그것은 별도의 PR에 있어야 합니다. 이것은 특히 관련없는 코드의 코드 스타일 변경을 포함합니다. PR을 열때, 만약 _Contributor License Agreement (CLA)_ 에 서명한 적이 없으시다면, 서명을 요청해야 함을 명심하세요. CLA[는 여기서 확인 할 수 있습니다.](https://contribute.meteor.com).

6.  **리뷰.** 적어도 한명의 핵심 기여자는 merge되어지기 전에 변화에 대한 서명을 해야 합니다. 코드 리뷰에 대한 중요한 것들을 배우기 위해선 아래의 "code reivew" 섹션을 확인하세요. 만약 merge될 코드를 더 빨리 처리하기 위해서, 먼저 여러분 스스로 자신의 코드를 리뷰해보세요.
7.  **Merge 와 release!**

### 코드 리뷰 가이드

Apollo 패키지들의 모든 코드들이 비슷한 codebase를 가지는 하나 이상의 기여자들에게 리뷰를 받는 것은 중요합니다. 여기 여러분들이 찾고 있는 몇 가지가 있습니다.

1.  **필수적인 CI 체크 통과** 이것은 리뷰를 하기 위한 전제 조건이고, PR 작성자의 책임입니다. 테스트들이 통과되지 않는 한, PR은 검토되지 않습니다.
2.  **간략함.** 의도된 목표를 달성하기 위한 가장 쉬운 방법이 맞나요? 불필요한 함수ㅗ들, 복잡한 라인들에 의해 너무 많은 파일이 있다면, 더 심플한 방법을 제안하세요. 특히 간단하고, 작고, 실용적인 개선을 할 수 있을 경우, 지나치게 일반적인 솔루션을 구현하는 것을 피하세요.
3.  **테스트하기.** 다른 코드들이 변화할 때, 이 코드가 깨지지 않을 것이라는 것을 확신하세요? 이것이 망가질 때, 추가된 테스트들이 우리에게 라이브러리의 어떤 부분이 문제가 되었는지 식별 할 수 있게 도움을 주나요? 우리가 외각에 있는 사례들을 적절하게 다루었나요? 하나라도 해당된다면, 테스트 커버리지 리포트를 보세요. 새 코드에서 모든 중요한 경로들이 최소한 한 번 이상 실행 되나요?
4.  **필수적이지 않고, 관련없는 변경사항.** PR들은 특히 코드와 관련없는 임의의 형식 변경이 포함되서는 안됩니다. 만약 리팩토링이 되어야 한다면, 가능한 버그 수정 혹은 기능 개선 PR과 분리되어야 합니다.
5.  **코드는 적절한 Comment를 가집니다.** 코드는 comment가 달려야 하거나, “self-documenting”방식에 의해 깔끔하게 작성되어야 합니다.
6.  **언어의 관용적 사용** Typescript를 통해, 입력내용이 명시적이고 정확한 것임을 확신하세요. ES2015를 통해 require 보다는 imports 사용하고, var 대신에 const를 사용하는 것등을 확실시 하세요. 일반적인 표현을 사용하고 주위의 코드 스타일을 따르세요.

## 새로운 contributors

Apollo server에 기여하길 원하지만, 어디서 시작해야 할지 모르겠다면, [로드맵과 디자인 문서](./ROADMAP.md)을 살펴보세요. 여러분이 흥미로운 것들중 하나를 선택 하세요. 그리고 작업하세요. 디자인 문서가 충분히 깔끔하지 않다면 (아마 아닐꺼에요), 우리가 토론 할 수 있게 이슈를 열어 주세요.

마지막으로, 수많은 친절한 기여자분들과 대화할 수 있는 곳인 [Apollo Spectrum community](https://spectrum.chat/apollo)에 가입하세요
